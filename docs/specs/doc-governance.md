# 文档治理清单（可执行）

版本：v1.0  日期：2026-04-29

## 1. 目标
- 保持 `docs/`、`agents/`、`skills/` 边界清晰，避免重复、过期和膨胀。
- 让“规则、分流、动作”三层结构长期可维护。

## 2. 目录职责（硬规则）
1. `docs/specs/core.md`
- 唯一规则源（SSOT）。
- 所有约束性规则只能在这里定义和修改。

2. `agents/README.md`
- 总执行入口。
- 仅定义执行顺序、任务分流、Skill 触发映射。

3. `skills/*`
- 仅定义操作步骤模板。
- 必须引用 `core.md`，不得独立定义规则。

## 3. 新增文档准入规则
1. 新增 `docs/specs/*` 前必须先判断能否并入 `core.md`。
2. 只有“领域独立且长期稳定”的内容才允许拆分为独立 spec。
3. 新增 `skills/*` 前必须先判断能否并入已有 skill。
4. 禁止新增“只有原则、没有步骤”的 skill 文档。

## 4. 变更同步规则
1. 任何配置字段、脚本路径、关键流程变化，必须同 PR 同步更新：
- `docs/specs/core.md`
- `agents/README.md`（若任务分流受影响）
- 对应 `skills/*`（若步骤受影响）
2. 若新增或删除 spec 文件，必须更新：
- `docs/specs/README.md`
- `docs/README.md`（当入口结构变化时）

## 5. 清理与合并规则
1. 连续 60 天未被引用且与现有 skill 重叠，进入合并/删除候选。
2. 语义重复（>70% 内容重叠）的 skill，保留一个主文档，其余合并删除。
3. 过期路径、失效命令、旧字段名必须在发现当次 PR 修复，不留“后续再改”。

## 6. 评审检查清单（PR 必查）
1. 规则是否只改在 `core.md`？
2. `agents` 是否只做分流，不重复规则？
3. `skills` 是否只写步骤，不写规则定义？
4. 文档中的路径和代码仓库当前实际路径是否一致？
5. 是否存在可删除的重复文档没有处理？

## 7. 建议执行节奏
1. 每两周做一次轻量文档巡检（路径有效性、重复检测、过期内容）。
2. 每月做一次结构收敛（合并低价值文档、删除未使用模板）。
